| Код | Требования | Комментарий |
|-----|------------|-------------|
| **F** | **Функциональные (Functionality)** | |
| F1 | Подача заявок на депозиты: сайт (ФИО, телефон) и интернет-банк (счет, сумма) | Базовая MVP-функциональность |
| F2 | Подача заявок на кредиты: сайт (ФИО, телефон, паспорт, предварительное решение) и интернет-банк (предодобренные предложения) | Требует интеграцию с системой скоринга |
| F3 | Отображение продуктов: общие условия на сайте, персонализированные в интернет-банке | Потребует кэширование для производительности |
| F4 | Обработка заявок бэк-офисом с ускоренной обработкой предодобренных кредитов в течение дня | Требует изменения интеграции с daily batch на near real-time |
| F5 | Управление депозитными ставками сотрудниками кредитного и депозитного бэк-офисов | Возможная реализация в АБС как расширение текущего процесса |
| F6 | СМС-уведомления для подтверждения операций и информирования о статусах заявок | Использует существующий СМС-функционал ядра |
| F7 | Идентификация новых клиентов в отделении после подачи заявки с сайта | Юридическое требование, влияет на сквозной процесс |
| F8 | Продолжение работы с предодобренной заявкой в отделении, а не создание новой | Требует единого идентификатора заявки на всех каналах |
| F9 | Подтверждение операций СМС-кодом в интернет-банке | Обеспечение безопасности действий клиента |
| **U** | **Удобство использования (Usability)** | |
| U1 | Консистентный UI с корпоративной системой дизайна и брендинговыми элементами | Требует единой UI-библиотеки или дизайн-системы |
| U2 | Персонализированные ставки и предодобренные предложения в интернет-банке | Повышает лояльность клиентов и конверсию |
| **R** | **Надёжность (Reliability)** | |
| R1 | Доступность интернет-банка и новых сервисов 99.9% в режиме 24/7 | Критично для цифровых каналов, требует отказоустойчивой инфраструктуры |
| R2 | Отказоустойчивость при сбоях ЦОД | Необходимо использовать существующий резервный ЦОД, активное переключение |
| R3 | Сохранность заявок при сбоях | Недопустима потеря данных на любом этапе подачи |
| **P** | **Производительность (Performance)** | |
| P1 | Отклик интерфейса в миллисекундах, решение проблемы загрузки справочников >1 сек | Влияет на API-дизайн, кэширование, оптимизацию БД, асинхронные операции |
| P2 | Изоляция от нагрузки на АБС (избежание прямого обращения к перегруженной БД АБС) | Требует промежуточного слоя данных, репликации или асинхронной интеграции |
| P3 | Горизонтальное масштабирование новых компонентов | Stateless-архитектура с балансировщиком нагрузки |
| P4 | Недопущение дополнительной нагрузки на систему скоринга в часы пик | Предрасчеты скорингов не должны выполняться в рабочее время |
| **S** | **Поддерживаемость (Supportability)** | |
| S1 | Техническая документация для сопровождения и расширения | Обеспечение развития системы силами внутренней команды |
| S2 | Мониторинг и логирование для диагностики | Критично для обеспечения требований R1, R2 и P1 |
| S3 | Реализация нового функционала СМС без доработок ядра подрядчиком | Требует создания отдельного сервиса уведомлений поверх существующего ядра |
| **+R** | **+ Ограничения (Restrictions)** | |
| +R1 | Шифрование трафика (HTTPS) для передачи чувствительных данных | Обязательное требование для финансовых организаций |
| +R2 | Соответствие законодательству о защите персональных данных | Влияет на архитектуру хранения, контроль доступа и аудит |
| +R3 | Приоритет существующих технологий (MS SQL, Oracle) и экспертизы (Java) | Ограничивает выбор компонентов и решений |
| +R4 | Переход на микросервисную архитектуру (начиная с депозитного функционала) | Стратегическое решение, определяющее IT-ландшафт на годы |
| +R5 | Несовместимость текущей платформы интернет-банка с Kafka | Требует обновления платформы или промежуточных решений (адаптеры) |
| +R6 | Избежание доработок ядра интернет-банка подрядчиком (реализация внутренними силами) | Влияет на распределение задач — необходим слой API или отдельные сервисы |
| +R7 | Обмен АБС <-> Кредитный конвейер только раз в сутки (не может быть ускорен) | Требует промежуточных хранилищ и асинхронных паттернов для F4 |
| +R8 | АБС масштабируется только вертикально | Ключевое ограничение, влияющее на всю архитектуру интеграций |
| +R9 | Идентификация новых клиентов только в отделении (юридическое требование) | Влияет на дизайн процесса, онлайн-открытие невозможно на MVP |
| +R10| Предодобренные кредиты требуют повторного скоринга при подаче заявки | Обеспечивает актуальность решения, но требует оптимизации запросов |
